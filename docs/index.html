Response Format

Every response follows this structure:

Standard Response Shape
{
  "data": { ... },
  "message": "Human-readable message or null",
  "error": "Error description or null"
}
All protected endpoints require: Authorization: Bearer <accessToken>
Access tokens expire after 15 minutes - use POST /api/auth/refresh to renew.

Request Requirements (all POST endpoints)

These rules apply to every request that carries a body.

Header: Content-Type: application/json
Body size limit: 10 kb - larger bodies are rejected with 413.
Body must be valid JSON - malformed JSON is rejected with 400 before the route handler runs.
Common Errors (apply to every POST endpoint)
StatusErrorWhen
400Invalid JSON in request bodyThe request body is not valid JSON (parse failed)
413Request body too largeThe request body exceeds 10 kb
500Internal server errorUnexpected server-side failure
These are global fallbacks. Endpoint-specific errors are listed under each endpoint below.

POST /api/auth/register
Public ⏱ Rate Limited

Create a new user account. New users are assigned the free plan (level 1). Rate limited to 3 requests / hour per IP.

Request Body
FieldTypeRequiredDescription
emailstring✅Unique email address
namestring✅Display name
passwordstring✅User password

{

    "id": 1,
    "email": "user@example.com",
    "name": "User Name",

    "plan": { "id": 1, "name": "مجاني", "level": 1 }

  "message": "Registration successful",
  "error": null
}

400email, name, and password are requiredMissing fields
409Email already in useDuplicate email
429Too many registration attempts, please try again laterRate limit exceeded (3/hour)

🔒 Auth ⏱ Rate Limited

Activate a subscription code. Upgrades the user's plan and creates a subscription record. Rate limited to 10 requests / 15 minutes per IP.

Request Body
FieldTypeRequiredDescription

      "id": 4,
      "userId": 2,
      "planLevel": 2,
      "startDate": "2026-04-19T04:09:47.241Z",
      "endDate": "2026-05-19T04:09:47.241Z",

    },
    "user": {
      "id": 2,
      "email": "user@example.com",
      "name": "User Name",
      "planExpiresAt": "2026-05-19T04:09:47.241Z",

    }

  "error": null

🔒 Concurrency guarantee: If the same code is activated by multiple concurrent requests, exactly one succeeds with 200 and all others receive 400 Code already used. The activation is atomic - no duplicate subscriptions are ever created.

Important Notes

Security: isCorrect is never sent in question choices - only revealed after submitting an attempt.
Security: passwordHash is never included in any API response.
Cron Job: Expired plans are automatically downgraded to free (level 1) daily at 00:00 Asia/Damascus.
Plan expiry - calendar-day boundary (by design): A user whose subscription ends mid-day (for example at 14:00) remains on their paid plan until the next daily cron run at 00:00 Asia/Damascus. This is an intentional product policy - do not treat a user remaining on a paid plan a few hours after their planExpiresAt timestamp as a bug.
Tokens: Refresh tokens are stored server-side in the database - stateless refresh is not supported.
Refresh error distinction: On POST /api/auth/refresh, the error Refresh token expired is distinct from Invalid or expired refresh token. The former means the server deleted the stored row and the client must force a full re-login. The latter means the JWT signature/payload failed verification.
